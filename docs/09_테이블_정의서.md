# 09. 테이블 정의서

## 1. 문서 목적
이 문서는 Legal-Mind-RAG에서 사용하는 테이블 구조를 정리한다.

본 문서는 두 범위를 함께 다룬다.
- **현재 구현 기준 테이블**
- **장기 설계 기준 테이블**

즉, 이 문서를 읽을 때는 "실제 코드에 이미 있는 테이블"과 "추후 확장을 위해 설계된 테이블"을 구분해야 한다.

---

## 2. 현재 저장소 기준 DB 상태
- 현재 설정 DB: SQLite
- 설정 위치: `app/core/config.py`
- 현재 코드 기준 주요 모델 파일: `app/models/*`
- 실제 DB 파일(기본값): 프로젝트 루트의 `sql_app.db`
- 현재는 마이그레이션 도구 없이 `Base.metadata.create_all()`로 테이블을 생성/갱신한다. (`app/main.py`)

현재 코드상 직접 확인되는 테이블은 아래와 같다.
- `chat_sessions`
- `messages`
- `answer_metas`
- `answer_traces`
- `users`
- `documents`
- `document_chunks`
- `document_chunk_embeddings`

## 3. 현재 구현 테이블 정의
> 이 섹션은 "현재 코드/SQLite 기준으로 실제 사용 중인 테이블"만 다룬다.
> 타입 표기는 모델의 논리 타입(SQLAlchemy)와 SQLite 실제 저장 타입이 다를 수 있어, 필요한 경우 함께 적는다.

### 3.1 `chat_sessions`
**용도**
- 하나의 상담 주제를 세션 단위로 관리한다.

| 컬럼명 | 타입 | NULL 허용 | 키 | 기본값 | 설명 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| `chat_session_id` | UUID (SQLite: `CHAR(32)`) | N | PK | `uuid.uuid4` | 상담 세션 식별자 |
| `user_id` | UUID (SQLite: `CHAR(32)`) | Y | FK | 없음 | 사용자 식별자, `users.user_id` 참조 |
| `title` | String | Y | - | 없음 | 상담 제목 |
| `category` | String | Y | - | 없음 | 상담 카테고리 |
| `risk_level` | String | Y | - | 없음 | 위험도 |
| `summary` | Text | Y | - | 없음 | 전체 대화 요약 |
| `is_deleted` | Boolean | N | - | `False` | 삭제 여부 |
| `created_at` | DateTime(timezone=True) | N | - | `func.now()` | 생성 시각 |
| `updated_at` | DateTime(timezone=True) | N | - | `func.now()` / update 시 갱신 | 수정 시각 |

**관계**
- 1:N with `messages`
- N:1 with `users` (문서상/관계 선언상 기준)

## 5. 테이블 간 핵심 관계

### 현재 구현 중심 관계
- `chat_sessions` 1:N `messages`
- `messages` 1:1 `answer_metas`
- `messages` 1:N `answer_traces`
- `messages` self reference `parent_message_id`
- `users` 1:N `chat_sessions` (현재는 `chat_sessions.user_id`가 NULL 허용)
- `documents` 1:N `document_chunks`
- `document_chunks` 1:1 `document_chunk_embeddings`

### 장기 확장 관계
- `users` 1:N `chat_sessions`
- `users` 1:1 `user_profiles`
- `companies` 1:N `user_profiles`
- `users` 1:N `user_lifecycles`
- `documents` 1:N `doc_hierarchies`
- `documents` 1:N `document_chunks`
- `doc_hierarchies` 1:N `document_chunks`

## 6. 구현 시 주의사항
- 현재 DB 설정은 SQLite이며, UUID는 SQLite에선 `CHAR(32)`로 저장된다. (모델 논리 타입은 `Uuid`)
- 현재는 마이그레이션 없이 `Base.metadata.create_all()`로 생성되므로, 컬럼 추가/삭제 같은 스키마 변경은 의도치 않게 누락될 수 있다.
- `answer_traces.chunk_id`는 현재 DB에서 FK가 걸려있지 않다. (문서 조각과의 참조 무결성이 필요하면 FK/인덱스 추가를 고려)
- 문서(장기 설계)에는 `VECTOR`, `ENUM`, `UUID` 등의 PostgreSQL 전제 타입이 섞여 있다. 현재 구현 스키마와 혼동되지 않도록, 3장(현재 구현)과 4장(장기 설계)을 분리해서 본다.

## 7. 한 문장 요약
현재 테이블 구조는 상담(세션/메시지/메타/트레이스) + 사용자 + 지식베이스(문서/청크/임베딩)까지 SQLite로 구현되어 있고, 나머지 확장 테이블은 장기 설계 단계에 있다.
