# 09. 테이블 정의서

## 1. 문서 목적
이 문서는 Legal-Mind-RAG에서 사용하는 테이블 구조를 정리한다.

본 문서는 두 범위를 함께 다룬다.
- **현재 구현 기준 테이블**
- **장기 설계 기준 테이블**

즉, 이 문서를 읽을 때는 "실제 코드에 이미 있는 테이블"과 "추후 확장을 위해 설계된 테이블"을 구분해야 한다.

---

## 2. 현재 저장소 기준 DB 상태
- 현재 설정 DB: SQLite
- 설정 위치: `app/core/config.py`
- 현재 코드 기준 주요 모델 파일: `app/models/chat.py`
- 현재 테이블 정의와 관계 선언은 재정비가 필요한 초안 상태에 가깝다.

현재 코드상 직접 확인되는 테이블은 아래와 같다.
- `chat_sessions`
- `messages`
- `answer_metas`

`answer_traces`는 문서상 설계되어 있으나 현재는 주석 상태로 남아 있다.

---

## 3. 현재 구현 테이블 정의

### 3.1 `chat_sessions`
**용도**
- 하나의 상담 주제를 세션 단위로 관리한다.

| 컬럼명 | 타입 | NULL 허용 | 키 | 기본값 | 설명 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| `chat_session_id` | UUID | N | PK | `uuid.uuid4` | 상담 세션 식별자 |
| `user_id` | UUID | N | FK | 없음 | 사용자 식별자, `users.user_id` 참조 |
| `title` | String | Y | - | 없음 | 상담 제목 |
| `category` | String | Y | - | 없음 | 상담 카테고리 |
| `risk_level` | String | Y | - | 없음 | 위험도 |
| `summary` | Text | Y | - | 없음 | 전체 대화 요약 |
| `is_deleted` | Boolean | N | - | `False` | 삭제 여부 |
| `created_at` | DateTime(timezone=True) | N | - | `func.now()` | 생성 시각 |
| `updated_at` | DateTime(timezone=True) | N | - | `func.now()` / update 시 갱신 | 수정 시각 |

**관계**
- 1:N with `messages`
- N:1 with `users` (문서상/관계 선언상 기준)

---

### 3.2 `messages`
**용도**
- 세션 안의 개별 발화를 저장한다.

| 컬럼명 | 타입 | NULL 허용 | 키 | 기본값 | 설명 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| `message_id` | Integer | N | PK | autoincrement | 메시지 식별자 |
| `chat_session_id` | UUID | N | FK | 없음 | 소속 세션 ID |
| `message_index` | Integer | N | - | 없음 | 세션 내 메시지 순서 |
| `role` | String | N | - | 없음 | 발화 주체 (`user`, `assistant`, `system`) |
| `content` | Text | N | - | 없음 | 메시지 본문 |
| `token_usage` | Integer | Y | - | 없음 | 토큰 사용량 |
| `response_time_ms` | Integer | Y | - | 없음 | 응답 시간(ms) |
| `parent_message_id` | Integer | Y | FK | 없음 | 부모 메시지 ID, self reference |
| `created_at` | DateTime(timezone=True) | N | - | `func.now()` | 생성 시각 |

**관계**
- N:1 with `chat_sessions`
- 1:1 with `answer_metas`
- 1:N with `answer_traces` (현재 코드에는 관계 선언만 존재)

---

### 3.3 `answer_metas`
**용도**
- AI 답변에 대한 메타 정보와 근거 요약을 저장한다.

| 컬럼명 | 타입 | NULL 허용 | 키 | 기본값 | 설명 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| `meta_id` | Integer | N | PK | autoincrement | 답변 메타 식별자 |
| `message_id` | Integer | N | FK | 없음 | 대상 AI 메시지 ID |
| `disclaimer` | Text | Y | - | 없음 | 면책 문구 |
| `applied_rule` | Text | Y | - | 없음 | 적용 법 조항 또는 규정 |
| `confidence_score` | Float | Y | - | 없음 | 답변 확신도 |
| `created_at` | DateTime(timezone=True) | N | - | `func.now()` | 생성 시각 |

**관계**
- N:1 with `messages`

---

### 3.4 `answer_traces` (설계 단계)
**용도**
- 어떤 문서 조각이 어떤 답변에 영향을 주었는지 추적한다.

**상태**
- 문서에는 존재
- 코드에는 주석 처리
- 관계 선언은 `Message.answer_traces`에 남아 있음

| 컬럼명 | 타입 | NULL 허용 | 키 | 설명 |
| :--- | :--- | :--- | :--- | :--- |
| `trace_id` | Integer | N | PK | 추적 식별자 |
| `message_id` | Integer | N | FK | 대상 메시지 ID |
| `step_order` | Integer | Y | - | 추론 단계 순서 |
| `logic_type` | String | Y | - | 추론 유형 |
| `chunk_id` | Integer | Y | FK | 참조한 문서 조각 |
| `relevance_score` | Float | Y | - | 관련성 점수 |
| `created_at` | DateTime | N | - | 생성 시각 |

## 5. 테이블 간 핵심 관계

### 현재 구현 중심 관계
- `chat_sessions` 1:N `messages`
- `messages` 1:1 `answer_metas`
- `messages` 1:N `answer_traces` (설계 단계)
- `messages` self reference `parent_message_id`

### 장기 확장 관계
- `users` 1:N `chat_sessions`
- `users` 1:1 `user_profiles`
- `companies` 1:N `user_profiles`
- `users` 1:N `user_lifecycles`
- `documents` 1:N `doc_hierarchies`
- `documents` 1:N `document_chunks`
- `doc_hierarchies` 1:N `document_chunks`

## 6. 구현 시 주의사항
- 현재 DB 설정은 SQLite지만, 모델 일부는 PostgreSQL UUID 타입을 사용하고 있다.
- `chat_sessions.user_id`는 `users.user_id`를 참조하지만 현재 저장소에는 `users` 모델이 보이지 않는다.
- `Message.answer_traces` 관계는 선언되어 있으나 실제 `AnswerTrace` 모델은 주석 처리 상태다.
- `app/models/chat.py`는 `app.database.session`의 `Base`를 import하지만 현재 저장소에는 해당 파일이 없다.
- 따라서 `app/models/chat.py`는 현재 상태 그대로 유지하기보다, 실제 사용할 DB 전략과 구현 범위에 맞춰 다시 정리하는 것이 맞다.

따라서 현재 테이블 정의를 실제 DB로 바로 사용하려면 모델 정합성 확인이 필요하다.

---

## 7. 한 문장 요약
현재 테이블 구조는 상담 세션과 메시지와 답변 메타를 중심으로 구현되어 있고, 나머지 사용자·문서·알림 테이블은 장기 설계 단계에 있다.
