# 02. 시스템 아키텍처

## 1. 이 문서가 설명하는 것
Legal-Mind-RAG의 아키텍처는 크게 두 층으로 이해할 수 있다.

1. **현재 MVP 기준 아키텍처**
   - 사용자의 질문을 받고
   - 관련 근거 문서를 찾고
   - 근거 포함 답변을 생성하고
   - 상담 이력을 저장하는 흐름

2. **장기 확장 아키텍처**
   - 회원 맞춤 진단
   - 알림
   - 리포트
   - 문서 계층 관리
   - 고도화된 벡터 검색

현재 이 프로젝트의 중심은 두 번째가 아니라 첫 번째다.

---

## 2. MVP에서 가장 중요한 시스템 역할
이 시스템은 단순히 질문에 답하는 챗봇이 아니다.

현재 MVP에서 시스템이 반드시 해야 하는 일은 아래와 같다.

1. 사용자의 질문을 안전하게 입력받는다.
2. 질문과 관련된 법령/가이드라인/판례를 찾는다.
3. 찾은 근거를 바탕으로 설명형 답변을 생성한다.
4. 답변에 출처와 면책 문구를 포함한다.
5. 상담 흐름을 세션과 메시지 단위로 저장한다.

즉, 아키텍처의 핵심은 "근거 기반 상담 흐름"을 안정적으로 만드는 것이다.

---

## 3. 현재 MVP 기준 전체 흐름

### ① 사용자 질문 입력
- 사용자가 상담 화면에서 질문을 입력한다.
- 질문은 하나의 상담 세션 안에서 메시지로 축적된다.

### ② 백엔드 수신 및 전처리
- FastAPI가 요청을 받는다.
- 필요하면 사용자 맥락이나 입력 정리를 수행한다.
- 이후 검색 가능한 질의로 전달한다.

### ③ 관련 문서 검색
- 질문과 관련된 법령, 가이드라인, 판례 조각을 검색한다.
- 검색 결과는 답변 생성의 근거가 된다.

### ④ AI 답변 생성
- 검색된 근거와 사용자 질문을 함께 모델에 전달한다.
- 모델은 설명형 답변을 만들고, 출처와 주의 문구를 포함한다.

### ⑤ 상담 기록 저장
- 세션 정보, 메시지, 답변 메타를 저장한다.
- 이후 같은 세션에서 문맥을 이어갈 수 있다.

---

## 4. 현재 구조를 이루는 계층

### [A] Client Layer
- 역할: 사용자의 질문 입력과 답변 확인
- 핵심 경험: 채팅 중심 인터페이스, 출처 확인, 상담 흐름 유지

### [B] API Layer
- 기술 방향: FastAPI
- 역할: 요청 수신, 응답 반환, 세션 단위 입출력 제어

### [C] Application / Service Layer
- 역할: 상담 로직 조합
- 포함되는 책임:
  - 질문 처리
  - 문서 검색 호출
  - 모델 프롬프트 구성
  - 응답 메타 정리

### [D] Data Layer
- 역할: 상담 기록 및 검색 기반 데이터 관리
- 현재 중심:
  - 상담 세션/메시지 저장
  - 답변 메타 저장
- 장기 확장:
  - 문서 원본/계층 구조 저장
  - 벡터 검색 최적화
  - 사용자 프로필/알림 관리

---

## 5. 현재 기술 방향
| 영역 | 현재 기준 | 설명 |
| :--- | :--- | :--- |
| API | FastAPI | 상담 요청 처리 |
| Data Model | SQLAlchemy, Pydantic | 스키마 및 DB 모델링 |
| LLM Orchestration | LangChain 중심 방향 | 검색 결과와 모델 호출 연결 |
| Relational DB | SQLite | 현재 개발 단계에서 사용 |
| Future DB | PostgreSQL | 확장 시 고려 |
| Vector Search | 장기적으로 `pgvector` 고려 | 현재는 설계 중심 |

---

## 6. 현재 구현과 장기 설계의 차이
문서 전체를 읽을 때 가장 중요하게 봐야 하는 부분이다.

### 현재 구현에 가까운 부분
- 상담 세션
- 메시지 저장
- 답변 메타 저장

### 아직 장기 설계에 가까운 부분
- 사용자 도메인 전체
- 기업/라이프사이클 관리
- 알림 시스템
- 리포트 생성
- 문서 계층 관리
- 완전한 벡터 저장소 구조

즉, 아키텍처 문서의 모든 요소가 현재 코드에 이미 존재한다고 보면 안 된다.

---

## 7. 설계 원칙
이 프로젝트의 아키텍처는 아래 원칙을 따른다.

1. **신뢰 가능한 응답이 우선이다.**
   - 답변 속도보다 근거 제시 가능성이 더 중요하다.

2. **상담 흐름을 먼저 완성한다.**
   - 회원, 알림, 리포트보다 질문-검색-응답-저장의 흐름이 먼저다.

3. **장기 확장을 막지 않는 구조를 유지한다.**
   - 지금은 작게 시작하되, 나중에 사용자 도메인과 문서 저장 구조를 붙일 수 있어야 한다.

4. **문서와 코드의 차이를 명확히 드러낸다.**
   - 설계도와 구현 상태를 섞어 쓰지 않는다.

---

## 8. 앞으로 확장될 수 있는 흐름
MVP 이후에는 아래 흐름이 추가될 수 있다.

- 사용자 프로필을 반영한 법 적용 범위 판단
- 개인 맞춤형 노무 이슈 알림
- 상담 내용을 요약한 리포트 생성
- 문서 원문 뷰어와 하이라이트 연동
- 전문가 상담 연계를 위한 브리핑 출력

이 확장들은 중요하지만, 현재 아키텍처의 1차 목표는 아니다.

---

## 9. 한 문장 요약
현재 Legal-Mind-RAG의 시스템 아키텍처는 "근로자의 질문을 받아, 관련 근거를 검색하고, 출처가 포함된 답변을 생성해 저장하는 상담 MVP 구조"로 이해하는 것이 가장 정확하다.
